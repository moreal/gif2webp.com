# UI 색차(Color Difference) & 명도 대비(Luminance Contrast) 가이드

본 문서는 디지털 UI/UX 디자인에서 색상 차이를 **인지 가능성, 가독성, 접근성** 관점에서
실무적으로 적용하기 위한 기준을 정리한다.

---

## 1. UI에서 색차를 사용하는 목적

색채학적 색차:
- 두 색이 물리적으로 다른가?

UI에서의 색차:
- 사용자가 빠르게, 안정적으로, 실수 없이 구분할 수 있는가?

따라서 UI에서는 **순수 색차(ΔE)만 사용하지 않고**
명도 대비, 시각 조건, 색각 다양성을 함께 고려한다.

---

## 2. 색차(ΔE) 기준

### 2.1 색차 공식

- 사용 권장: **CIEDE2000 (ΔE₀₀)**
- 기준 색공간: **CIELAB (L\*, a\*, b\*)**

---

### 2.2 UI에서의 실무적 ΔE₀₀ 기준

| 사용 목적 | 권장 ΔE₀₀ | 설명 |
|---------|----------|------|
| 미세 변화 (Hover, Focus 등) | 3 ~ 5 | 의식하지 않아도 인지되는 상태 변화 |
| 의미 구분 (Primary / Secondary 등) | 7 ~ 10 | 한눈에 다른 역할로 인식 |
| 상태 / 경고 / 오류 | ≥ 10 | 즉각적 주의 환기 |

> ΔE₀₀ < 2 는 UI에서 사용하기에는 인지 안정성이 낮음

---

## 3. 명도 대비 (WCAG 기준)

UI 가독성에는 색차보다 **명도 대비가 더 큰 영향**을 미친다.

### 3.1 WCAG 대비 기준

| 요소 | 최소 명도 대비 |
|----|---------------|
| 일반 텍스트 | 4.5 : 1 |
| 큰 텍스트 (≥18pt 또는 Bold 14pt) | 3 : 1 |
| 아이콘 / UI 그래픽 | 3 : 1 |

> WCAG는 색차(ΔE)를 사용하지 않고 명도 대비만을 기준으로 삼는다.

---

## 4. 색차와 명도 대비의 관계

- ΔE가 커도 명도 대비가 낮으면 읽기 어렵다
- 명도 대비가 충분하면 ΔE가 작아도 구분 가능
- UI에서는 반드시 **두 기준을 동시에 만족**해야 한다

---

## 5. UI 컴포넌트별 색상 설계 가이드

### 5.1 버튼 상태 변화

| 상태 변화 | 권장 전략 |
|---------|-----------|
| Default → Hover | L\* +5 ~ +8 |
| Default → Active | L\* −8 ~ −12 |
| Disabled | 채도 감소 + 명도 대비 하락 |

- Lab 색공간에서 **L\* 중심 조정** 권장
- 결과 ΔE₀₀는 보통 4 ~ 6 수준

---

### 5.2 카테고리 / 태그 색상

- 서로 다른 Hue 사용
- ΔE₀₀ ≥ 8 확보
- L\*는 50 ~ 70 범위 유지 → 가독성 안정

---

## 6. 색각 이상(Color Vision Deficiency) 대응 원칙

- Hue 차이만으로 의미를 구분하지 말 것
- 반드시 다음 중 2가지 이상을 함께 사용
  - 명도 차이
  - 형태
  - 아이콘
  - 텍스트

---

## 7. UI 색상 설계 체크리스트

- [ ] ΔE₀₀ 기준 충족 (용도에 맞는 범위)
- [ ] WCAG 명도 대비 충족
- [ ] Hover / Active 상태가 즉시 인지되는가
- [ ] 색각 이상 환경에서도 의미가 유지되는가
- [ ] 어두운 배경 / 밝은 배경 모두에서 테스트했는가

---

## 8. 요약

- UI에서는 "구분 가능 최소치"보다 **인지 안정성 기준**을 사용한다
- 실무 권장:
  - 상태 변화: ΔE₀₀ 3~5
  - 의미 구분: 7~10
  - 경고/오류: 10 이상
- **항상 WCAG 명도 대비와 함께 적용**
